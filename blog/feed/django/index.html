I needed a way to keep my patches organized and updated. So, in this article I discuss my experiences with a tool I found to help me achieve these goals...</p> <p>Enter <a class="reference external" href="http://bazaar-vcs.org/">Bazaar</a>, one of the several <a class="reference external" href="http://en.wikipedia.org/wiki/Version_control_system#Distributed_version_control">distributed version control systems</a> that have been getting some hype recently. With <a class="reference external" href="http://en.wikipedia.org/wiki/Comparison_of_revision_control_software">so many to choose from</a>, why did I choose Bazaar? Well, maybe it was because of the cool name, or maybe it was the pretty website, or maybe just because it's written in the language I love—<a class="reference external" href="http://www.python.org/">Python</a>. If you are familiar with <a class="reference external" href="http://subversion.tigris.org/">Subversion</a>, then you will feel right at home with Bazaar. This is because Bazaar uses most of the same commands as Subversion (e.g. <tt class="docutils literal">svn ci = bzr ci</tt>, <tt class="docutils literal">svn st = bzr st</tt>, etc.).</p> <p>Following <a class="reference external" href="http://jameswestby.net/tips/tips/hacking-a-project-with-bzr.html">this guide for hacking an open source project using bzr</a>, I created a <tt class="docutils literal">django</tt> directory to stash my django hacks. After using svn to check out the latest Django code to a directory named <tt class="docutils literal">upstream</tt>, and turning <tt class="docutils literal">upstream</tt> into a bzr repository too, it was time to start the real work. To re-iterate in command line speak, here is what I did:</p> <p><strong>Update:</strong> The steps below have been updated to reflect changes with bzr commands (versions &gt;=0.15rc2).</p> <pre class="literal-block"> mkdir django cd django bzr init-repo . # use &quot;bzr init-repo --trees .&quot; here instead if using a version &lt;0.15rc2 svn co http://code.djangoproject.com/svn/django/trunk/ upstream cd upstream bzr init echo &quot;.svn&quot; &gt; .bzrignore # do not store svn metadata in our bzr repo bzr add bzr ci -m 'Initial import of Django.' </pre> <p>Usually, when you want to add a new feature to some code, you create a branch. The purpose of these, so called, feature branches is to keep the features separated. Trying to create multiple features in the same branch would just be a big mess. Creating branches in a Subversion repository is easy enough, you just <tt class="docutils literal">svn cp</tt> the code to a new location. It's the keeping your branch in sync with the trunk that's not as easy.</p> <p>With Subversion, you have to keep track of the revisions that have already been merged to your branch and make sure to only merge the changes that haven't already been merged. Not so with Bazaar; Bazaar keeps track of this for you. A simple <tt class="docutils literal">bzr merge</tt> will pull in upstream changes that have not been merged yet.</p> <p>So, when I wanted to start working on a patch for <a class="reference external" href="http://code.djangoproject.com/ticket/2473">ticket #2473</a>, I created a branch:</p> <pre class="literal-block"> bzr branch upstream in-and-empty-list-2473 </pre> <p>After <tt class="docutils literal">cd</tt>'ing into my new branch and committing my changes, I created a diff to attach to the ticket with the command:</p> <pre class="literal-block"> bzr diff -r branch:../upstream </pre> <p>So far, working with Bazaar has been a breeze. About a month ago (in <a class="reference external" href="http://code.djangoproject.com/changeset/3360">[3360]</a>), an <tt class="docutils literal">is_authenticated</tt> method was added to the User and AnonymousUser classes. These are the classes used for Django's default authentication system.</p> <p>Previously, the template code used for displaying content based on whether or not a user had authenticated went something like:</p> <div class="highlight"><pre><span></span><span class="cp">{%</span> <span class="k">if</span> <span class="k">not</span> <span class="nv">user.is_anonymous</span> <span class="cp">%}</span><span class="x"></span> <span class="x">Content for logged in users.</span> <span class="cp">{%</span> <span class="k">else</span> <span class="cp">%}</span><span class="x"></span> <span class="x">Content for non-logged in users.</span> <span class="cp">{%</span> <span class="k">endif</span> <span class="cp">%}</span><span class="x"></span> </pre></div> <p>The problem with this code is that if, somehow, the <tt class="docutils literal">user</tt> template variable did not get populated, your template would treat the requesting user as if they were logged in.</p> <p>Notice that this sort of behavior will be seen for <strong>all</strong> negative if statements. The template code <tt class="docutils literal">{% if not variable %}</tt> will always evaluate to <tt class="docutils literal">True</tt> if <tt class="docutils literal">variable</tt> doesn't exist in the context. This is because Django uses the <a class="reference external" href="http://www.djangoproject.com/documentation/settings/#template-string-if-invalid">settings.TEMPLATE_STRING_IF_INVALID</a> value for non-existent template variables; by default, <tt class="docutils literal">TEMPLATE_STRING_IF_INVALID</tt> is <tt class="docutils literal">''</tt>, which evaluates to <tt class="docutils literal">False</tt> (and <tt class="docutils literal">not False</tt> is <tt class="docutils literal">True</tt>).</p> <p>Notice also that even using the template code:</p> <div class="highlight"><pre><span></span><span class="cp">{%</span> <span class="k">if</span> <span class="nv">user.is_anonymous</span> <span class="cp">%}</span><span class="x"></span> <span class="x">Content for non-logged in users.</span> <span class="cp">{%</span> <span class="k">else</span> <span class="cp">%}</span><span class="x"></span> <span class="x">Content for logged in users.</span> <span class="cp">{%</span> <span class="k">endif</span> <span class="cp">%}</span><span class="x"></span> </pre></div> <p>(without the <tt class="docutils literal">not</tt>) will not work in this case because if the <tt class="docutils literal">user</tt> variable is non-existent, the requesting user will still be treated as if they were authenticated.</p> <p>The new, recommended way for checking that a requesting user has been authenticated in your templates is:</p> <div class="highlight"><pre><span></span><span class="cp">{%</span> <span class="k">if</span> <span class="nv">user.is_authenticated</span> <span class="cp">%}</span><span class="x"></span> <span class="x">Content for logged in users.</span> <span class="cp">{%</span> <span class="k">else</span> <span class="cp">%}</span><span class="x"></span> <span class="x">Content for non-logged in users</span> <span class="cp">{%</span> <span class="k">endif</span> <span class="cp">%}</span><span class="x"></span> </pre></div> <p>Here, if the <tt class="docutils literal">user</tt> template variable were to not exist, your template would treat the requesting user as non-authenticated, just as we would want.</p>
